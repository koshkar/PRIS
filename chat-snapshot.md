## Шаг 1 — Изучение контекста из предыдущих лабораторных

Прочитаны файлы `Preview.wsd` (диаграмма компонентов) и `ЛР5.docx` (Activity + Sequence диаграммы).

**Ключевые артефакты архитектуры MediConnect из ЛР3–5:**

- **Акторы:** Пациент, Врач, Администратор клиники
- **Мобильное приложение:** UI Auth / МедКарта / Передача данных, локальный кэш (offline), Клиент API
- **Серверные сервисы:** API Gateway, Auth Service (OAuth2/OIDC), Medical Records Service, Consent/Access Control Service, Notification Service, Audit/Log Service
- **Интеграции:** FHIR/HL7 Adapter, ESB / Message Broker → МИС клиники, ЛИС, Страховая система
- **БД:** PostgreSQL (Patient/Records)

**Ключевые бизнес-правила из ЛР5:**
- Проверка токена сессии при запуске (offline-кэш при отсутствии сети)
- **Обязательное согласие** пациента перед любой передачей данных врачу
- Сохранение согласия + уведомление врачу после передачи

## Шаг 2 — Создание прототипа

### Структура файлов

```
mediconnect-prototype/
├── index.html          # разметка + HTML <template>-шаблоны трёх ролей
├── style.css           # CSS с переменными, responsive grid (1100px / 760px)
├── app.js              # роутер + эмуляция 5 сервисов поверх localStorage
├── README.md           # инструкция запуска и демо-аккаунты
├── seed_patient.html   # быстрый вход как пациент (для демо)
├── seed_doctor.html    # быстрый вход как врач
├── seed_admin.html     # быстрый вход как администратор
└── screenshots/
    ├── 00-login.png
    ├── 01-patient.png
    ├── 02-doctor.png
    └── 03-admin.png
```

### Структура данных

| Сущность | Поля |
|---|---|
| User | id, login, pwd, role, name, dob, allergies, conditions, specialty |
| Record | id, patientId, type (lab/visit/prescription), date, text |
| Consent | id, patientId, doctorId, granted, ts |
| Notification | id, doctorId, patientId, recordId, ts, seen |
| AuditEvent | id, ts, actorId, action, target, meta |

### Демо-аккаунты

| Роль | Логин | Пароль |
|---|---|---|
| Пациент | `ivanov` | `1234` |
| Пациент | `petrova` | `1234` |
| Врач | `smirnov` | `1234` |
| Врач | `kuznetsova` | `1234` |
| Администратор | `admin` | `1234` |

### Реализованные сценарии

1. **Auth с проверкой роли** — попытка войти с «чужой» ролью блокируется и логируется (`login_role_mismatch`)
2. **Медкарта пациента** — ФИО, дата рождения, аллергии, хронические заболевания, список анализов и приёмов
3. **Управление согласиями** — выдача и отзыв согласия конкретному врачу в один клик, метка времени
4. **Передача данных врачу** — без активного согласия блокируется (`share_blocked_no_consent`), с согласием создаётся уведомление
5. **Кабинет врача** — список пациентов с согласием, уведомления «Новое» / «Просмотрено», создание записей о приёме
6. **Кабинет администратора** — журнал аудита с текстовым фильтром, реестр пользователей, статус 5 сервисов
